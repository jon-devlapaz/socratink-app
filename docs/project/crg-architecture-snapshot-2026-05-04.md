## TL;DR

socratink-app is a **115-file FastAPI + vanilla-JS codebase** with **1455 nodes / 10166 edges** in the graph. Three structural facts dominate:

1. **The frontend is in two giant JS files.** `public/js/app.js` (**4126 lines**) and `public/js/graph-view.js` (**2767 lines**, with a 1441-line `mountKnowledgeGraph` function inside) hold most of the visible surface. Any UI work touches one of them.
2. **The Python backend is more modular than the frontend.** `ai_service.py` (1044 lines) is the AI hub; `auth/router.py` (979) + `main.py` (843) + `auth/service.py` (431) form the request-handling spine. Smaller modules cluster cleanly under `auth/`, `source_intake/`, `learning_commons/`, `llm/`, `models/`, `admin/`.
3. **Test density is high** — 414 test nodes (28% of all nodes). 1826 `TESTED_BY` edges. The "high coupling" CRG warnings between `tests` and various source communities are **not smells** — they're tests doing their job.

Most-depended-on symbols (caller count): `auth/service.py::AuthSessionState` (32), `admin/todo_parser.py::parse_tink_todo` (25), `source_intake/parse.py::extract_html` (17), `auth/service.py::AuthUser` (17). Touching these without `get_impact_radius_tool` first is how silent regressions ship.

## What CRG can't see (be honest)

The graph is structural truth for what Tree-sitter can parse. It does **not** capture:

- **Frontend↔backend wire**: `public/js/api-client.js` calls `/api/extract` over HTTP. There's no static call edge from JS to the Python handler in the graph. They show up as separate communities (`js-node` vs `socratink-app-request`) with zero edges between them, which is technically correct but operationally misleading.
- **`app_prompts/*.txt`**: load-bearing for AI behaviour but invisible to CRG (text files, not code). Use Claude Context for that surface.
- **Runtime registration / dynamic dispatch**: anything that resolves a handler via string lookup at runtime. FastAPI's decorator-based routes mostly get captured, but if `ai_service.py` has any drill-type → handler mapping by string, those edges may be missing.
- **Vercel deployment surface**: `vercel.json` config and the `api/index.py` entrypoint shape what actually runs in prod, but they're invisible to the graph beyond raw imports.
- **Vector similarity in `semantic_search_nodes_tool`** is currently dormant — 1340 Google embeddings are stored but query-time code defaults to a local provider that isn't installed (CRG v3.2.4 bug, see `todo.md`). FTS5 keyword search still works and is empirically strong on symbol queries.

The "graph counts are a floor, not a ceiling" memory applies to all of the above — verify "single call site" or "no test coverage" claims with `rg` before acting.
